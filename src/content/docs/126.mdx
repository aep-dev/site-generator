# Enumerations

It is common for a field to only accept or provide a discrete and limited set
of values. In these cases, it can be useful to use enumerations (generally
abbreviated "enums") in order to clearly communicate what the set of allowed
values are.

## Guidance

APIs <b class="font-extrabold text-green-700">may</b> expose enum objects for sets of values that are expected to change
infrequently:

```typescript
// Possible formats in which a book may be published.
enum Format {
  // The printed format, in hardback.
  Hardback = 'HARDBACK',

  // The printed format, in paperback.
  Paperback = 'PAPERBACK',

  // An electronic book format.
  Ebook = 'EBOOK',

  // An audio recording.
  Audiobook = 'AUDIOBOOK',
}
```

- All enum values <b class="font-extrabold	text-yellow-700">should</b> use a consistent case format across an
  organization. In many cases, this is dictated by the IDL the organization
  uses.
- Enums <b class="font-extrabold	text-yellow-700">should</b> document whether the enum is frozen or they expect to add
  values in the future.

### When to use enums

Enums can be more accessible and readable than strings or booleans in many
cases, but they do add overhead when they change. Therefore, enums <b class="font-extrabold	text-yellow-700">should</b>
receive new values infrequently. While the definition of "infrequently" may
change based on individual use cases, a good rule of thumb is no more than once
a year. For enums that change frequently, the API <b class="font-extrabold	text-yellow-700">should</b> use a string and
document the format.

<Aside type="note" title="Note">
  If an enumerated value needs to be shared across APIs, an enum
  <b class="font-extrabold text-green-700">may</b> be used, but the assignment between enum values and their wire
  representation <b class="font-extrabold	text-red-700">must</b> match.
</Aside>

### Alternatives

Enums <b class="font-extrabold	text-yellow-700">should not</b> be used when there is a competing, widely-adopted standard
representation (such as with [language codes][bcp-47] or [media types][]).
Instead, that standard representation <b class="font-extrabold	text-yellow-700">should</b> be used. This is true even if
only a small subset of values are permitted, because using enums in this
situation often leads to frustrating lookup tables when trying to use multiple
APIs together.

For enumerated values where the set of allowed values changes frequently, APIs
<b class="font-extrabold	text-yellow-700">should</b> use a `string` field instead, and <b class="font-extrabold	text-red-700">must</b> document the allowed
values. String fields with enumerated values <b class="font-extrabold	text-yellow-700">should</b> use a uniform case
system (`snake_case`, `kebab-case`, etc.) throughout an organization.

Boolean fields <b class="font-extrabold text-green-700">may</b> be used in situations where it is clear that no further
flexibility will be needed. The default value <b class="font-extrabold	text-red-700">must</b> be `false`.

### Compatibility

Adding values to an enum has the potential to be disruptive to existing
clients. Consider code written against the `Format` enum in an earlier version
where only the first two options were available:

```typescript
switch (book.format) {
  case Format.Hardback:
    // Do something...
    break;
  case Format.Paperback:
    // Do something...
    break;
  default:
    // When new enum values are introduced, pre-existing client code may
    // throw errors or act in unexpected ways.
    throw new Error('Unrecognized value.');
}
```

Services <b class="font-extrabold text-green-700">may</b> add new values to existing enums; however, they <b class="font-extrabold	text-yellow-700">should</b> add
enums carefully; think about what will happen if a client system does not know
about a new value.

Additionally, in IDLs where enum values are presented in a specific order,
services <b class="font-extrabold	text-yellow-700">should</b> only add new values to the end. An exception to this rule
is if the enum conforms to an external standard (for example, an enum
representing HTTP status codes would add a new 3xx value alongside the others,
not at the end).

## Interface Definitions


<Tabs>
  <TabItem label="Protocol Buffers">
  
  
  <Sample path="../aep.dev/aep/general/0126/enum.proto" type="protobuf" />
  
  - The zero value of the enum <b class="font-extrabold	text-yellow-700">should</b> be the name of the enum itself followed
    by the suffix `_UNSPECIFIED`. The service <b class="font-extrabold text-green-700">may</b> either allow or prohibit
    use of this value.
  - Enums which will only be used in a single message <b class="font-extrabold	text-yellow-700">should</b> be nested within
    that message. In this case, the enum <b class="font-extrabold	text-yellow-700">should</b> be declared immediately
    before it is used.
  - If multiple enums are in the same namespace, they <b class="font-extrabold	text-red-700">must not</b> share any
    values. (This is because enums do not provide their own namespace for their
    values in some languages.)
  - If an enumerated value needs to be shared across APIs, an enum <b class="font-extrabold text-green-700">may</b> be
    used, but the assignment between the value names and the tag numbers <b class="font-extrabold	text-red-700">must</b>
    match.
  
  **Note:** When using protocol buffers, it is impossible to distinguish between
  `false` and unset. If this is a requirement, an enum <b class="font-extrabold text-green-700">may</b> be a better design
  choice (although `google.protobuf.BoolValue` is also available).
  
  
  </TabItem>
  <TabItem label="OpenAPI 3.0">
  
  
  <Sample path="../aep.dev/aep/general/0126/enum.oas.yaml" type="yml" />
  
  - Enumerated fields <b class="font-extrabold	text-yellow-700">should</b> be strings.
  - If the enum is optional, The `null` value <b class="font-extrabold	text-yellow-700">should</b> be used as the empty
    value, and <b class="font-extrabold	text-yellow-700">should</b> be the first value specified.
  
  **Note:** If `null` is a valid value, OpenAPI 3.0 also requires that
  `nullable: true` is specified for the field.
  
  
  </TabItem>
</Tabs>
