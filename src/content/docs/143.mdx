Many common concepts, such as spoken languages, countries, currency, and so on,
have common codes (usually formalized by the [International Organization for
Standardization][iso]) that are used in data communication and processing.
These codes address the issue that there are often different ways to express
the same concept in written language (for example, "United States" and "USA",
or "Espa√±ol" and "Spanish").

## Guidance

For concepts where a standardized code exists and is in common use, fields
representing these concepts <b class="font-extrabold	text-yellow-700">should</b> use the standardized code for both input
and output.


<Tabs>
  <TabItem label="Protocol Buffers">
  
  
  ```proto
  // A message representing a book.
  message Book {
    // Other fields...
  
    // The IETF BCP-47 language code representing the language in which
    // the book was originally written.
    // https://en.wikipedia.org/wiki/IETF_language_tag
    string language_code = 99;
  }
  ```
  
  
  </TabItem>
  <TabItem label="OpenAPI 3.0">
  
  
  ```json
  
  {
    "type": "object",
    "properties": {
      "language_code": {
        "type": "string",
        "description": "The IETF BCP-47 language code
          representing the language in which the book was originally written. See
          https://en.wikipedia.org/wiki/IETF_language_tag"
      }
    }
  }
  
  ```
  
  
  </TabItem>
</Tabs>
    

- Fields representing standardized concepts <b class="font-extrabold	text-red-700">must</b> use the appropriate data
  type for the standard code (usually `string`).
  - Fields representing standardized concepts <b class="font-extrabold	text-yellow-700">should not</b> use enums, even if
    they only allow a small subset of possible values. Using enums in this
    situation often leads to frustrating lookup tables when using multiple APIs
    together.
  - Fields representing standardized concepts <b class="font-extrabold	text-red-700">must</b> indicate which standard
    they follow, preferably with a link (either to the standard itself, the
    Wikipedia description, or something similar).
- The field name <b class="font-extrabold	text-yellow-700">should</b> end in `_code` or `_type` unless the concept has an
  obviously clearer suffix.
- When accepting values provided by users, validation <b class="font-extrabold	text-yellow-700">should</b> be
  case-insensitive unless this would introduce ambiguity (for example, accept
  both `en-gb` and `en-GB`). When providing values to users, APIs <b class="font-extrabold	text-yellow-700">should</b>
  use the canonical case (in the example above, `en-GB`).

### Content types

Fields representing a content or media type <b class="font-extrabold	text-red-700">must</b> use [IANA media types][].
For legacy reasons, the field <b class="font-extrabold	text-yellow-700">should</b> be called `mime_type`.

### Countries and regions

Fields representing individual countries or nations <b class="font-extrabold	text-red-700">must</b> use the [Unicode
CLDR region codes][cldr] ([list][]), such as `US` or `CH`, and the field
<b class="font-extrabold	text-red-700">must</b> be called `region_code`.

<Aside type="caution" title="Important">
  We use `region_code` and not `country_code` to include regions
  distinct from any country, and avoid political disputes over whether or not
  some regions are countries.
</Aside>

### Currency

Fields representing currency <b class="font-extrabold	text-red-700">must</b> use [ISO-4217 currency codes][iso-4217],
such as `USD` or `CHF`, and the field <b class="font-extrabold	text-red-700">must</b> be called `currency_code`.

<Aside type="note" title="Note">
  For representing an amount of money in a particular currency, rather
  than the currency code itself, use the [aep type.money schema][money]
</Aside>

### Language

Fields representing spoken languages <b class="font-extrabold	text-red-700">must</b> use [IETF BCP-47 language
codes][bcp-47] ([list][]), such as `en-US` or `de-CH`, and the field <b class="font-extrabold	text-red-700">must</b>
be called `language_code`.

### Time zones

Fields representing a time zone <b class="font-extrabold	text-yellow-700">should</b> use the [IANA TZ][] codes, and the
field <b class="font-extrabold	text-red-700">must</b> be called `time_zone`.

Fields also <b class="font-extrabold text-green-700">may</b> represent a UTC offset rather than a time zone (note that
these are subtly different). In this case, the field <b class="font-extrabold	text-red-700">must</b> use the [ISO-8601
format][] to represent this, and the field <b class="font-extrabold	text-red-700">must</b> be named `utc_offset`.
